# Chapter 3: Exploratory Data Analysis

This chapter presents an exploratory data analysis (EDA) for the Amazon Reviews 2023 dataset in the Books category [10, 4][cite: 1]. The analysis serves two goals[cite: 1]. First, it characterizes the data distribution and quality of the raw sources and of the working subset used in this project[cite: 1]. Second, it translates those observations into concrete design decisions for multimodal retrieval, including text truncation policies, image fallback behaviors, and ranking calibration under strong positive bias[cite: 1].

The dataset provides two complementary streams: a review stream containing user-generated text and optional user-uploaded images describing personal experience and subjective judgment, and an item metadata stream offering catalog-side descriptions and cover images that describe the product itself [10][cite: 1]. We analyze both streams separately and then jointly after merging them by `parent_asin`[cite: 1]. All plots referenced in this chapter are generated by our EDA notebook and saved as figures[cite: 1].

## 3.1 Dataset Overview

### 3.1.1 Amazon Reviews 2023: Books Category

**Source and collection methodology.**
Amazon Reviews 2023 is a large-scale public benchmark that pairs product metadata with user reviews [10, 4][cite: 1]. Each review references an item via `asin` and `parent_asin`, enabling item-level aggregation and retrieval evaluation[cite: 1]. For the Books category, the dataset includes both textual descriptions and high coverage of catalog images (book covers), which makes it suitable for multimodal retrieval and recommendation experiments [10][cite: 1]. The dataset reflects real-world e-commerce scenarios where users interact with products through multiple modalities, though the distribution of these modalities varies significantly as we will demonstrate[cite: 1].

**Overall scale.**
In the raw Books split used in this project, the dataset contains 29,475,453 review records and 4,448,181 item metadata records [10][cite: 1]. These numbers motivate sampling for rapid iteration and for feasible preprocessing on shared compute[cite: 1]. The full dataset presents challenges in terms of memory footprint, preprocessing time, and embedding computation that would require distributed infrastructure beyond the scope of this capstone project[cite: 1]. Furthermore, preliminary analysis suggested that the distributional patterns we seek to understand stabilize well before examining the complete corpus, making strategic sampling a practical choice[cite: 1].

**Sampling strategy for this project.**
To balance compute limits and diversity, we sample from the raw corpus by extracting `SAMPLE_REVIEWS` = 600,000 review records and `SAMPLE_META` = 400,000 metadata records[cite: 1]. This sampling is performed randomly to avoid temporal or popularity bias[cite: 1]. After filtering records with missing critical fields, joining by `parent_asin`, and removing short text pairs that would not provide sufficient semantic content, the working merged dataset used for most EDA plots in this chapter contains 76,347 merged records spanning 38,654 unique books and 34,048 unique users[cite: 1]. This merged subset is large enough to reveal stable distributional patterns while remaining manageable for repeated experimentation[cite: 1]. The reduction from initial samples to final merged records reflects the reality that not all reviews successfully join with metadata, and that quality filtering is necessary to ensure meaningful text fields for embedding models[cite: 1].

### 3.1.2 Data Schema

**Review features.**
Each review record includes several fields that capture both the user's opinion and metadata about the review itself [10][cite: 1]. The `rating` field stores a floating-point star rating in the range 1 to 5, representing the user's overall assessment[cite: 1]. The review text content is split between `title` and `text` fields, where the title typically contains a short summary and the text contains the detailed commentary[cite: 1]. An optional `images` field stores a list of dictionaries, each containing URLs for user-uploaded images at multiple resolutions (`large_image_url`, `medium_image_url`, and `small_image_url`)[cite: 1]. The `user_id` provides an anonymized reviewer identifier, while `timestamp` records when the review was posted as a UNIX timestamp in milliseconds[cite: 1]. Additional credibility signals include `helpful_vote`, which counts how many users found the review helpful, and `verified_purchase`, a boolean indicating whether Amazon confirmed the reviewer purchased the item[cite: 1].

**Item metadata features.**
Each metadata record contains catalog-side information for a book, including standard bibliographic fields such as `title`, `author`, and `categories` [10][cite: 1]. The `features` field typically contains bullet-point marketing descriptions, while `description` provides longer narrative text about the book's content or purpose[cite: 1]. Commercial information appears in the `price` field[cite: 1]. Cover images are stored in an `images` dictionary containing lists of URLs under keys `hi_res`, `large`, and `thumb`, offering multiple resolution options[cite: 1]. The `details` field stores structured attributes as JSON-formatted text, capturing information such as publisher, language, ISBN, and physical dimensions[cite: 1]. This heterogeneous schema reflects the practical reality of e-commerce data, where different fields serve different purposes in the product catalog[cite: 1].

**Data quality assessment.**
Across the merged working subset, core modeling fields such as rating, review text, and item text are effectively complete after filtering[cite: 1]. Our preprocessing pipeline discards records where these critical fields are missing or contain only whitespace, ensuring that the remaining data supports meaningful semantic analysis[cite: 1]. However, image availability differs sharply between modalities, creating a multimodal imbalance that must be handled explicitly during training and evaluation [10][cite: 1]. We quantify this gap in Section 3.4[cite: 1]. Beyond completeness, we also observe variation in text quality, with some reviews containing informal language, typos, or highly subjective expressions, while item descriptions tend toward more standardized marketing language[cite: 1].

## 3.2 Text Length Distribution

Text length directly impacts model input limits, batching efficiency, and truncation policy[cite: 1]. Understanding the distribution of text lengths allows us to make informed decisions about how much content to preserve and where to apply truncation strategies that balance information retention against computational constraints[cite: 1].

In this project we define review text as the concatenation of `title` and `text` with a separator token, creating a unified representation that captures both the summary and detailed commentary[cite: 1]. We define item text as a constructed description from catalog fields, primarily drawing from `title`, `features`, and `description`, with the option to include `categories` and `author` information depending on the experiment[cite: 1]. These constructions reflect common practices in information retrieval where multiple fields are combined to create richer semantic representations [7][cite: 1].

We analyze length in word counts for EDA purposes, as words provide an intuitive unit for understanding content volume[cite: 1]. Later, we translate these distributions into token budget decisions for transformer encoders [21], accounting for the fact that subword tokenization typically produces 1.3 to 1.5 tokens per word depending on vocabulary coverage and text domain [1][cite: 1].

### 3.2.1 Review Text Analysis

The review length distribution is strongly right-skewed, following a pattern common in user-generated content where most contributions are brief but a minority of users write extensively[cite: 1]. Figure 3.1 visualizes this distribution, showing a sharp peak at shorter lengths and a gradual tail extending to several thousand words in extreme cases[cite: 1].

In the merged dataset, the mean review length reaches 86.2 words while the median sits at just 37.0 words[cite: 1]. This substantial gap between mean and median indicates that a small fraction of extremely long reviews pulls the average upward while the typical review remains quite concise[cite: 1]. The distribution reveals that approximately half of all reviews contain fewer than 40 words, often consisting of just a sentence or two expressing overall sentiment[cite: 1]. These brief reviews still carry signal through their ratings and sparse keywords, but they provide limited semantic content for retrieval compared to more detailed entries[cite: 1].

At the upper end of the distribution, we observe outliers extending beyond 1,000 words, with the maximum reaching 4,013 words in a single review[cite: 1]. These lengthy entries sometimes represent thoughtful literary analysis or detailed comparisons across editions, but they can also include tangential personal narratives or off-topic digressions[cite: 1]. For modeling purposes, the extreme tail presents a practical challenge because including full text would create batch heterogeneity and computational inefficiency[cite: 1].

**Implication for BLAIR input limits.**
Transformer encoders such as BLAIR are typically evaluated with a 512-token input window, which corresponds roughly to 350-400 words after accounting for tokenization overhead [4][cite: 1]. The observed distribution suggests that the vast majority of reviews fit comfortably within common token limits once tokenized[cite: 1]. Specifically, reviews below the 90th percentile require no truncation, while only the longest 10% of reviews exceed typical model capacities[cite: 1]. This motivates a consistent truncation rule that preserves the beginning of each review, where users typically state their main point, while discarding the tail[cite: 1]. Such a policy maintains comparability across training and evaluation by ensuring all inputs conform to the same length constraints, and it aligns with findings from prior work showing that truncating from the end of documents preserves most semantic content for retrieval tasks[cite: 1].

The presence of very short reviews also raises questions about minimum length thresholds[cite: 1]. Reviews containing only a few words may not provide sufficient context for meaningful semantic matching, suggesting that filtering out reviews below some word count (such as 10 or 15 words) could improve overall retrieval quality by removing noise[cite: 1]. We explore this trade-off through ablation studies in subsequent chapters[cite: 1].

### 3.2.2 Item Description Analysis

Item descriptions exhibit a different length profile compared to reviews[cite: 1]. In the merged subset, item text achieves a mean length of 51.9 words and a median of 50.0 words[cite: 1]. The near-equality of mean and median indicates a more symmetric distribution with fewer extreme outliers, which is expected because catalog text is curated by publishers, authors, or Amazon itself rather than generated freely by users[cite: 1]. Professional editorial standards and platform guidelines constrain description length and style, producing more consistent formatting across items[cite: 1].

The tighter distribution around 50 words reflects the typical structure of book descriptions in e-commerce settings, where marketers aim to convey essential information without overwhelming potential buyers[cite: 1]. Descriptions generally include the book's genre or subject matter, a brief plot summary or thematic overview, and occasionally author credentials or endorsements[cite: 1]. This compact format serves browsing users well but may omit details that would help with precise retrieval for niche queries[cite: 1].

Despite this overall consistency, item text can still become long when multiple fields contain verbose lists or repeated marketing phrases[cite: 1]. For example, the `features` field sometimes includes several dozen bullet points highlighting selling points, while the `description` field may contain multiple subsections of promotional text[cite: 1]. When all these fields are concatenated without processing, the resulting text can exceed several hundred words[cite: 1]. Additionally, certain categories such as textbooks or reference works tend toward longer descriptions that explain the book's structure, intended audience, and key topics covered[cite: 1]. The maximum item length in our subset reaches 1,250 words, though such cases are rare[cite: 1].

**Trade-off between completeness and token limits.**
Adding more metadata fields can improve recall for rare queries by including additional keywords and concepts that users might reference in their reviews[cite: 1]. For instance, including author names helps match reviews that mention the author's other works or writing style, while category information can resolve ambiguity for books with generic titles[cite: 1]. However, expanding the item text also increases noise and token usage, potentially diluting the signal from core descriptive content[cite: 1].

Based on these distributions, a compact construction combining title with a selection of short features and a truncated description offers a practical balance for BLaIR embeddings [4][cite: 1]. This approach captures the essential bibliographic information and marketing highlights without exceeding token budgets or introducing excessive promotional repetition[cite: 1]. Richer text constructions that include the full feature list or multiple description subsections can be explored through ablation experiments in Chapter 4, where we measure their impact on retrieval metrics[cite: 1]. Such ablations help quantify whether the additional content improves semantic coverage enough to justify the increased computational cost and potential noise[cite: 1].

## 3.3 Rating Distribution

Ratings provide a coarse sentiment signal and strongly affect retrieval evaluation, because highly rated items dominate the corpus[cite: 1]. Understanding the rating distribution is essential for interpreting retrieval results and for designing evaluation protocols that account for the skewed nature of user feedback in e-commerce settings[cite: 1].

### 3.3.1 Overall Rating Patterns

In Books, ratings follow a pronounced J-shaped distribution where 5-star reviews are most common, followed by 4-star reviews, while lower ratings appear progressively less frequently[cite: 1]. This pattern reflects a combination of factors including self selection bias (satisfied customers are more motivated to leave positive reviews), the nature of book purchasing (buyers often research before purchasing, leading to more satisfied customers), and potential platform effects where negative experiences may lead users to simply not review rather than leave harsh feedback[cite: 1].

Specifically, the rating distribution in our merged subset shows approximately 58% of reviews at 5 stars, 22% at 4 stars, 10% at 3 stars, 6% at 2 stars, and 4% at 1 star[cite: 1]. This heavy skew toward positive ratings creates several implications for modeling[cite: 1]. First, if ratings were used as a primary signal for relevance, the system would exhibit limited discriminative power since the majority of items cluster at the top of the scale[cite: 1]. Second, random negative sampling for contrastive learning may yield pairs where both the positive and negative items have high ratings, reducing the quality of the training signal[cite: 1]. Third, evaluation metrics that rely on rating based relevance labels may conflate genuine semantic similarity with rating distributions, making it difficult to isolate the model's ability to match content based queries[cite: 1].

The consistency of this pattern across different books, users, and time periods suggests it is a stable characteristic of the domain rather than a sampling artifact[cite: 1]. Books may naturally attract positive ratings because readers self select titles aligned with their interests and because reading a book represents a significant time investment that creates commitment bias[cite: 1].

### 3.3.2 Rating and Text Length Relationship

We examine whether review length correlates with rating to understand if sentiment intensity or polarization affects verbosity[cite: 1]. Figure 3.2 shows the distribution of review lengths segmented by rating score[cite: 1]. The median review length remains relatively stable across all rating levels, typically falling between 35 and 45 words[cite: 1]. This suggests that rating alone does not strongly predict how much users write, challenging the intuitive notion that extreme opinions (either very positive or very negative) would consistently prompt longer commentary[cite: 1].

However, the variance within each rating group differs notably[cite: 1]. Five-star and one-star reviews exhibit wider spreads and more extreme outliers compared to mid-range ratings[cite: 1]. This pattern indicates that while the typical review length is similar regardless of rating, strongly held opinions (whether positive or negative) occasionally inspire detailed elaboration[cite: 1]. Some highly satisfied readers write extensive praise describing exactly what they loved, while some disappointed readers write lengthy critiques explaining specific issues[cite: 1]. In contrast, three-star reviews tend toward more moderate lengths, possibly because ambivalent opinions require less justification or generate less emotional investment in explanation[cite: 1].

From a modeling perspective, this finding suggests that combining rating and length as features would provide limited additional signal beyond using each independently[cite: 1]. The weak correlation also means that length-based filtering (such as removing very short reviews) will not systematically bias the rating distribution, allowing us to apply quality thresholds without inadvertently affecting the balance of positive and negative feedback[cite: 1].

### 3.3.3 Implications for Retrieval

The extreme positive bias in ratings presents both challenges and opportunities for retrieval system design[cite: 1]. On one hand, ratings cannot serve as a primary relevance signal because they lack discriminative power when most items cluster at 4 or 5 stars[cite: 1]. A retrieval system that simply ranks items by average rating would fail to capture the semantic relationship between query and item content[cite: 1]. On the other hand, the abundance of high-quality positive reviews provides rich textual data for learning semantic embeddings, since users writing positive reviews often describe book content in detail rather than simply expressing dissatisfaction[cite: 1].

The rating distribution also informs evaluation choices[cite: 1]. Standard retrieval metrics typically assume binary or graded relevance judgments, but in our setting, nearly all item-review pairs that match semantically will also have high ratings[cite: 1]. This makes it difficult to use ratings as ground truth for relevance without additional processing[cite: 1]. One approach is to define relevance based on whether a review explicitly references a book rather than using rating thresholds[cite: 1]. Another approach is to focus on ranking metrics that measure relative ordering rather than absolute relevance, accepting that most retrieved items will be positively rated[cite: 1].

For training contrastive models like BLAIR, [4] the positive bias suggests that random negatives may not provide sufficient difficulty[cite: 1]. If both positive and negative examples in a contrastive pair have similar ratings, the model must learn to distinguish them based on subtle semantic differences rather than obvious sentiment gaps[cite: 1]. This could actually improve model quality by forcing the system to attend to content rather than relying on coarse sentiment features, though it may require larger batch sizes or harder negative mining strategies to maintain training signal strength[cite: 1].

**Truncation policy for model inputs.**
Based on the observed text length distributions and the constraints of transformer architectures, [21] we adopt the following truncation policies for our retrieval pipeline[cite: 1]. For review text, we truncate to 512 tokens to match BLAIR's input capacity, [4] keeping the beginning of the review where users typically state their main point and core opinions[cite: 1]. This policy balances information retention against computational efficiency, and it aligns with the finding that most reviews naturally fit within this limit[cite: 1]. For item text, we construct compact descriptions by concatenating title and selected features, then truncating the description field to preserve the first several sentences while staying within token budgets[cite: 1]. This retains core bibliographic attributes and marketing highlights while avoiding noisy repetition common in lengthy promotional text[cite: 1].

These truncation decisions are evaluated empirically in Chapter 4 through ablation studies that vary the token limits and field selections, measuring the impact on retrieval metrics such as recall and mean reciprocal rank[cite: 1]. The goal is to establish a policy that maintains strong retrieval performance while ensuring consistent preprocessing across all items and queries[cite: 1].

## 3.4 Image Availability

A central question for multimodal retrieval is how often each modality is available and whether missing modalities create systematic gaps in coverage[cite: 1]. Books presents an unusual case in that catalog cover images are nearly universal while user-uploaded review images are rare, creating a stark asymmetry that affects how we design the multimodal retrieval pipeline [10][cite: 1].

### 3.4.1 Review Images

In the merged dataset, only 1.8% of reviews include at least one user uploaded image, corresponding to 1,337 reviews out of 76,347 total [10][cite: 1]. This sparse coverage reflects the nature of book reviews, where textual commentary suffices for most users purposes[cite: 1]. Unlike product categories such as electronics or clothing where images of the actual received item (showing packaging, assembly, fit, etc.) add substantial value, book reviews typically focus on content, writing quality, and subjective enjoyment, aspects more naturally expressed through text[cite: 1].

When user images do appear, they often show physical aspects such as book condition, printing quality, page layout, or comparisons between different editions[cite: 1]. For example, a user might photograph damaged pages received from a third-party seller, or compare the size and binding quality of a hardcover versus paperback edition[cite: 1]. These images can provide valuable evidence for specific types of queries related to physical product quality, but their sparsity severely limits their utility as a primary retrieval signal[cite: 1].

From a modeling perspective, the 1.8% coverage means that training image based encoders on review images would leave the vast majority of data unutilized[cite: 1]. Furthermore, the distribution of image availability may correlate with specific review types (such as complaints about condition) rather than representing a balanced sample across all content[cite: 1]. This selection bias makes it risky to draw general conclusions from the small subset of reviews with images[cite: 1]. Consequently, our retrieval pipeline treats review images as an optional qualitative enhancement rather than a core component, and we do not attempt to build specialized embeddings for this modality within the scope of this project[cite: 1].

### 3.4.2 Item Cover Images

In sharp contrast to review images, 99.1% of items in the merged subset have a valid cover image URL, demonstrating near-universal coverage for this modality [10][cite: 1]. This high availability stems from Amazon's catalog infrastructure, where book cover images are standard product metadata sourced directly from publishers or ISBN databases[cite: 1]. The remaining 0.9% of items without cover images likely represent edge cases such as very old catalog entries, self-published works uploaded without proper metadata, or formatting errors in the source data[cite: 1].

The near complete coverage of cover images makes CLIP-based visual embeddings broadly applicable at the catalog level [14][cite: 1]. Every retrieval query can potentially leverage visual similarity to book covers, providing a complementary signal to text based matching[cite: 1]. Cover images encode information that may not be fully captured in textual descriptions, including visual design elements, series branding, genre conventions, and aesthetic tone[cite: 1]. For instance, a fantasy novel might have elaborate illustrated artwork that signals its target audience, while an academic textbook might feature a minimalist design with disciplinary color schemes[cite: 1]. These visual cues can help disambiguate items with similar titles or generic descriptions[cite: 1].

However, cover images also have limitations[cite: 1]. Different editions of the same book often have completely different covers, which can create false negatives in retrieval if a user's mental image comes from one edition but the catalog shows another[cite: 1]. Additionally, cover design trends evolve over time, meaning that older printings may have visual styles that differ substantially from contemporary releases[cite: 1]. The visual similarity captured by CLIP may not always align with semantic relevance, particularly for abstract or minimalist cover designs that convey little about content [14][cite: 1].

**Fallback strategy.**
Given the multimodal asymmetry between text (universally available), item cover images (nearly universal), and review images (rare), we design a retrieval pipeline that degrades gracefully when modalities are missing[cite: 1]. The primary approach uses multimodal scoring by combining BLaIR text embeddings with CLIP visual embeddings through late fusion or weighted averaging [4, 14][cite: 1]. When an item cover image exists, we compute both text and visual similarity scores and combine them according to learned or heuristic weights[cite: 1]. When an item lacks a cover image, the system falls back to text only retrieval using BLAIR embeddings exclusively, [4] ensuring that all items remain accessible even if imperfectly represented[cite: 1].

Review images, when present, are reserved for qualitative analysis and specialized experiments rather than integrated into the main retrieval pipeline[cite: 1]. Future work could explore using review images for specific tasks such as filtering by physical condition or providing visual evidence in retrieval explanations, but the core recommendation system does not depend on them[cite: 1]. This design choice reflects a pragmatic assessment of data availability and the trade offs between model complexity and coverage[cite: 1].

## 3.5 Category and Genre Analysis

The Books category encompasses a vast range of genres, subgenres, and topical classifications that often overlap in complex ways[cite: 1]. Many items share similar titles, cover aesthetics, and keywords across different subgenres, making strict categorical filtering brittle and potentially unreliable for retrieval tasks[cite: 1].

### 3.5.1 Category Overlap and Ambiguity

The `categories` field in item metadata contains hierarchical tags such as "Fiction > Science Fiction > Space Opera" or "Nonfiction > History > Military History > World War II." [10][cite: 1]. While these categories provide useful organizational structure, they suffer from several limitations[cite: 1].

First, many books legitimately belong to multiple categories but may only be tagged with one or two, creating incomplete coverage[cite: 1]. For example, a historical novel might reasonably appear under both "Fiction > Historical" and "History > European History," but catalog constraints often force a single primary categorization[cite: 1].

Second, category boundaries are inherently fuzzy in literature[cite: 1]. Is a book about the philosophy of science categorized under "Science" or "Philosophy"? Does a memoir by a war veteran belong in "Biography," "Military History," or "Psychology"?[cite: 1]. These edge cases reveal that categorical labels impose artificial discrete boundaries on a fundamentally continuous space of book content and themes[cite: 1]. User queries often reflect this continuity, asking for books "similar to $X$" without caring about precise genre labels[cite: 1].

Third, category hierarchies vary in granularity across different subdomains[cite: 1]. Some genres like "Romance" or "Mystery" have extensive subdivisions (paranormal romance, cozy mysteries, police procedurals), while others like "Poetry" remain relatively flat[cite: 1]. This inconsistency makes it difficult to apply uniform category-based filtering rules across the entire corpus[cite: 1].

### 3.5.2 Visual Ambiguity in Cover Design

Cover design conventions create additional challenges for categorical distinction[cite: 1]. Publishers and designers follow genre specific visual templates to signal target audience and set appropriate expectations[cite: 1]. For instance, thrillers often feature dark colors, bold typography, and urban imagery, while young adult fantasy tends toward illustrated figures and whimsical fonts[cite: 1]. However, these conventions overlap across related genres, making visual similarity an imperfect proxy for content similarity [14][cite: 1].

Series books and editions present particular difficulties[cite: 1]. Books in the same series typically share consistent design elements (matching fonts, color schemes, layout structure) to promote brand recognition[cite: 1]. While this helps readers identify related titles, it also means that CLIP embeddings may cluster books from the same series based on visual similarity even when their content differs significantly [14][cite: 1]. Similarly, classic works reprinted by the same publisher often receive uniform cover treatments (such as Penguin Classics' consistent design language), creating visual similarity across thematically diverse titles[cite: 1].

Reissues and anniversary editions further complicate the picture[cite: 1]. A novel originally published with a photographic cover might later receive an illustrated cover for a new edition, or vice versa[cite: 1]. From CLIP's perspective, these are visually distinct items despite representing identical content[cite: 1]. Conversely, commemorative editions of different books might share similar anniversary branding, creating false visual connections[cite: 1].

### 3.5.3 Implications for Modeling

The observed category ambiguity and visual overlap motivate a retrieval approach based on continuous semantic embeddings rather than discrete categorical filtering [7][cite: 1]. By learning dense vector representations through BLAIR (for text) and CLIP (for images), [4, 14] the system can capture gradations of similarity that reflect the actual complexity of book content and user preferences[cite: 1]. A book that blends historical fiction with elements of mystery can naturally appear in relevant positions for queries spanning both genres without requiring explicit category membership[cite: 1].

Embedding based retrieval also handles the long tail of niche topics more gracefully than categorical systems [7][cite: 1]. A query about "books discussing the intersection of neuroscience and mindfulness" might not map cleanly to any single category, but embedding similarity can identify relevant titles by matching key concepts in the text regardless of their formal classification[cite: 1]. This flexibility comes at the cost of interpretability (it is harder to explain why a particular book was retrieved based on embedding similarity than based on category matching), but for practical recommendation purposes the improved coverage and recall may justify this trade-off[cite: 1].

The reliance on semantic similarity does introduce new challenges[cite: 1]. Embeddings may conflate superficial textual similarity (such as books that use similar vocabulary or writing style) with genuine topical relevance[cite: 1]. They may also struggle with highly specific factual queries that require exact keyword matching rather than fuzzy semantic approximation[cite: 1]. These limitations suggest that embeddings work best as a primary ranking signal supplemented by auxiliary filters or reranking stages that can apply stricter criteria when needed[cite: 1].

## 3.6 Temporal Patterns

Temporal analysis examines how review activity and item availability evolve over time, helping identify potential biases in the dataset and informing decisions about evaluation protocols[cite: 1].

### 3.6.1 Review Distribution Over Time

We analyze review timestamps by converting the millisecond UNIX time into calendar dates and aggregating reviews by month and year[cite: 1]. The temporal distribution reveals several patterns worth noting[cite: 1]. First, review volume increases over time in the raw dataset, reflecting Amazon's growth and the expansion of the Books category [10][cite: 1]. This means that more recent items and reviews are over-represented in random samples, potentially biasing retrieval toward contemporary titles unless corrective measures are taken[cite: 1].

Second, we observe seasonal spikes corresponding to holiday shopping periods and back to school cycles[cite: 1]. Review activity peaks in late November through December (gift purchases) and again in August through September (textbook and academic book purchases)[cite: 1]. These spikes create temporal clustering where certain items receive concentrated attention during specific periods, which could affect evaluation if test sets are drawn from these high activity windows[cite: 1].

Third, the distribution of review ages varies across items[cite: 1]. Popular titles accumulate reviews continuously over many years, while niche titles may have sparse review activity concentrated around their release period[cite: 1]. This variation means that the temporal context of a review (how long ago it was written relative to the item's publication) differs systematically across popularity tiers[cite: 1].

From a modeling perspective, temporal effects matter primarily for evaluation design[cite: 1]. Random sampling of reviews for training or testing can over-represent recent popular items, leading to optimistic performance estimates that may not generalize to older catalog entries or less popular books[cite: 1]. While the main retrieval task in this project is content-based rather than time aware, understanding temporal patterns helps us design more balanced evaluation splits and identify potential confounds[cite: 1].

Future extensions of this work could incorporate time aware sampling to ensure diverse representation across publication years, or could evaluate whether retrieval accuracy degrades for older editions and out of print items[cite: 1]. Temporal drift in writing style and vocabulary might also affect model performance, particularly if training data comes predominantly from recent reviews while evaluation targets historical content[cite: 1].

### 3.6.2 Correlation Analysis

For numerical Feature Correlations, we compute pairwise correlations between numerical features to assess whether any features are redundant or strongly predictive of others[cite: 1]. The primary features examined include review rating, review text length (in words), item text length, helpful votes, and timestamp[cite: 1]. Understanding these correlations helps validate that the dataset contains genuine multivariate structure rather than collapsing into a single dominant dimension[cite: 1].

The analysis reveals that most feature pairs exhibit weak to moderate correlations[cite: 1]. Review rating and review length show a correlation of approximately 0.15, indicating a slight tendency for longer reviews to have more extreme ratings (both high and low), consistent with the variance patterns observed in Section 3.3.2[cite: 1]. However, this correlation is too weak to support using length as a proxy for sentiment or vice versa[cite: 1].

Helpful votes correlate weakly with review length (approximately 0.25), suggesting that longer reviews have some advantage in accumulating helpful votes, possibly because they provide more detailed information that users find valuable[cite: 1]. However, many short reviews also receive high helpful vote counts when they concisely express widely shared sentiments, preventing this relationship from being deterministic[cite: 1].

Item text length shows minimal correlation with review characteristics (below 0.1 for most pairs), confirming that catalog descriptions are relatively independent of user feedback patterns [10][cite: 1]. This independence supports the multimodal fusion approach, since text and metadata sources provide complementary information rather than redundant signals[cite: 1].

Timestamp correlates weakly with several other features due to temporal trends in the platform (for example, more recent reviews may be slightly longer on average as users become more engaged with online reviewing), but these correlations remain below 0.2, indicating that time effects do not overwhelm other sources of variation[cite: 1].

The modest correlation structure validates the use of multiple features for retrieval and recommendation [7][cite: 1]. If features were highly correlated (above 0.8), it would suggest redundancy where simpler models using fewer inputs could achieve comparable performance[cite: 1]. The observed pattern instead indicates that combining rating, text, and metadata captures distinct aspects of the review-item relationship[cite: 1].

## 3.7 Key Insights for System Design

For the embedding complementarity, the EDA reveals a natural division of labor between text and visual modalities that informs our multimodal architecture[cite: 1]. BLAIR embeddings capture semantic content from review and item text, including author names, edition attributes, thematic keywords, plot elements, and stylistic descriptions [4][cite: 1]. This textual signal provides the primary semantic matching capability, identifying items whose content aligns with user interests expressed in review queries[cite: 1].

CLIP embeddings capture visual aesthetics from book covers, encoding design elements such as color schemes, typography, illustration style, and genre specific visual conventions [14][cite: 1]. These visual features serve several complementary roles[cite: 1]. First, they help separate items with similar titles but different editions, where visual design may signal different target audiences or publishing contexts[cite: 1]. Second, they encode genre and tone information that may not be fully articulated in text, such as the distinction between literary fiction (minimalist, artistic covers) and commercial fiction (dramatic, high contrast covers)[cite: 1]. Third, they provide a tie breaker for ambiguous text matches, where multiple items have similar textual descriptions but different visual presentations[cite: 1].

This complementarity motivates late fusion scoring strategies where text embeddings establish the primary ranking and visual embeddings refine the ordering[cite: 1]. Specifically, we can compute separate similarity scores for text and images, then combine them through weighted averaging or learned fusion functions[cite: 1]. The relative weights can be tuned based on validation set performance, allowing the system to balance semantic matching against visual coherence depending on which signal proves more reliable in practice[cite: 1].

An alternative early fusion approach, where text and images are combined before computing similarity, could also be explored but may lose the ability to degrade gracefully when modalities are missing[cite: 1]. Late fusion preserves modularity and interpretability, making it easier to diagnose which modality contributes to retrieval decisions and to adjust the system when data characteristics change[cite: 1].

**Data Quality for Multimodal Retrieval**
Several practical conclusions follow from the EDA that directly shape system implementation[cite: 1].

The analysis confirms sufficient text coverage across both reviews and items, with core text fields effectively complete after minimal filtering [10][cite: 1]. This enables strong text-only baselines using BLAIR embeddings, [4] ensuring that the system provides reasonable retrieval performance even before incorporating visual signals[cite: 1].

The high cover availability at 99.1% means that CLIP based visual embeddings can be applied to nearly all items without requiring extensive fallback logic [10, 14][cite: 1]. This broad coverage supports using images as a reliable auxiliary modality rather than a specialized feature available only for premium items[cite: 1]. The small fraction of items without covers can be handled through simple text-only fallback without significantly compromising overall system coverage[cite: 1].

The sparse review images at 1.8% coverage dictate that the system cannot depend on user-uploaded images for general retrieval [10][cite: 1]. While these images may be valuable for specific use cases such as condition assessment or visual evidence in reviews, they do not form part of the core retrieval pipeline[cite: 1]. This design choice reflects a pragmatic assessment that building infrastructure for a signal present in fewer than 2% of cases introduces complexity without proportional benefit[cite: 1].

The positive bias in ratings, with approximately 80% of reviews at 4 or 5 stars, has multiple implications[cite: 1]. It suggests that ratings alone cannot serve as effective relevance signals for ranking, since most items cluster at the top of the scale[cite: 1]. It motivates careful negative sampling strategies for contrastive learning, [4] ensuring that negative examples are sufficiently distinct from positives in semantic content rather than just rating[cite: 1]. It also informs evaluation choices, where we must define relevance based on content match rather than rating thresholds[cite: 1].

Together, these findings support a retrieval architecture that prioritizes text embeddings as the primary signal, augments them with visual embeddings where available, and designs evaluation protocols that account for the skewed nature of e-commerce feedback[cite: 1]. The system can achieve reasonable performance through text alone while using images to improve precision for ambiguous queries[cite: 1].

## 3.8 Chapter Conclusion

This chapter analyzed the Books subset of Amazon Reviews 2023 at both the raw scale of millions of records and the working scale of approximately 76,000 merged records used for detailed analysis [10][cite: 1]. The working subset contains 38,654 unique books and 34,048 unique users, with high completeness in core fields after quality filtering[cite: 1]. The sampling and filtering process reduced the dataset to a manageable size while preserving the distributional patterns necessary for informed modeling decisions[cite: 1].

Three primary findings directly inform the system design presented in subsequent chapters[cite: 1].
*   First, text length distributions for both reviews and item descriptions are strongly right-skewed, with most content fitting comfortably within transformer token limits [21, 1] but a long tail of verbose entries requiring explicit truncation policies[cite: 1]. We adopt a strategy of truncating from the end to preserve the beginning of each text, where key information typically appears, while maintaining consistent input lengths across all samples[cite: 1].
*   Second, ratings exhibit extreme positive bias with approximately 80% of reviews at 4 or 5 stars, following the J-shaped distribution common in e-commerce[cite: 1]. This skew indicates that ratings cannot serve as primary relevance signals and that evaluation protocols must account for the imbalanced distribution[cite: 1]. The weak correlation between rating and text length confirms that these features provide independent information rather than redundant signals[cite: 1].
*   Third, image availability shows a stark asymmetry between modalities[cite: 1]. Item cover images achieve 99.1% coverage, supporting broad application of CLIP visual embeddings across the catalog [10, 14][cite: 1]. In contrast, user-uploaded review images appear in only 1.8% of cases, too sparse to serve as a primary retrieval signal [10][cite: 1]. This asymmetry motivates a multimodal architecture that combines universal text embeddings with near-universal visual embeddings while gracefully degrading to text-only retrieval when covers are missing[cite: 1].

Additional findings on category ambiguity, temporal patterns, and feature correlations provide supporting context for design decisions[cite: 1]. The overlap and fuzziness of genre categories justify embedding-based retrieval over strict categorical filtering [7][cite: 1]. Temporal analysis reveals seasonal spikes and recency bias that inform evaluation split design[cite: 1]. Correlation analysis confirms that numerical features provide complementary information without problematic redundancy[cite: 1].

Together, these insights establish a data-driven foundation for the model selection, fusion strategies, and evaluation protocols described in Chapter 4[cite: 1]. By grounding design decisions in observed data characteristics rather than assumptions, we aim to build a retrieval system that handles the practical complexities and imperfections inherent in real-world e-commerce data[cite: 1].
